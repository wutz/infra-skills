# XSKY XEOS 对象存储规划工具

帮助用户根据容量和性能需求，规划 XSKY XEOS 对象存储部署方案，计算所需服务器数量、选择合适的纠删码方案、确定最优磁盘配置。

## 概述

此 skill 会计算：
- 满足容量需求所需的服务器台数
- 合适的纠删码方案（EC8+2 或 EC4+2）
- 最优的磁盘大小配置
- 预估的性能指标（带宽和 OPS）

## 用户输入需求

### 1. 容量需求（必填）

询问用户的容量需求，支持十进制和二进制单位：
- **十进制单位**：TB、PB（1000 进制）
- **二进制单位**：TiB、PiB（1024 进制）—— **推荐使用**

始终向用户推荐使用二进制单位（TiB、PiB），因为它们与实际存储计算更匹配。

### 2. 性能需求（可选）

询问用户是否有性能需求，针对 4MB 大小对象的操作：
- **上传带宽**：例如 100 Mbps、1 Gbps、100 MB/s、1 GB/s
- **下载带宽**：例如 200 Mbps、2 Gbps、200 MB/s、2 GB/s
- **上传 OPS**：例如 1000 IOPS（针对 4K 对象）
- **下载 OPS**：例如 3000 IOPS（针对 4K 对象）

带宽单位支持比特制（Mbps、Gbps）和字节制（MB/s、GB/s）。

## 硬件配置

每台服务器的固定配置如下：

### 计算与内存
- **处理器**：2 × Intel Xeon 4314（2.4GHz，16核/32线程，135W）
- **内存**：8 × 32GB DDR4 = 256GB 总容量
- **系统盘**：2 × 960GB SATA SSD（RAID1）
- **索引缓存盘**：4 × 1.6TB NVMe SSD（读写均衡型，≥3 DWPD）

### 数据盘
- **默认配置**：每台服务器 32 × 16TB HDD
- **可选磁盘大小**：8TB、10TB、12TB、18TB、20TB、22TB、24TB（根据性能需求选择）

## 性能基准（单盘）

使用以下数值估算集群总性能：
- **上传带宽**：30 MiB/s 每盘
- **下载带宽**：60 MiB/s 每盘
- **上传 OPS**：100 IOPS 每盘（4K 对象）
- **下载 OPS**：300 IOPS 每盘（4K 对象）

## 计算流程

按以下步骤依次执行：

### 步骤 1：将容量统一转换为 TiB

将所有用户输入的容量转换为 TiB，以便统一计算：
- 如果输入是 TB：`TiB = TB × 0.909`
- 如果输入是 PB：`TiB = PB × 1000 × 0.909`
- 如果输入是 TiB：直接使用
- 如果输入是 PiB：`TiB = PiB × 1024`

使用 Bash 工具进行精确计算。

### 步骤 2：计算满足容量需求的服务器台数（优先使用 EC8+2）

首先尝试使用高效率的 EC8+2 纠删码：

```
所需台数 = 需求容量(TiB) / (单盘容量(TB) × 0.909) / 32 / 0.81 / 0.8
```

其中：
- `单盘容量(TB) × 0.909`：将 TB 转换为 TiB（例如 16TB × 0.909 = 14.544 TiB）
- `32`：每台服务器的磁盘数量
- `0.81`：空间保留和损耗（预留 19%）
- `0.8`：EC8+2 可用率（80% 效率）

向上取整到最接近的整数。

**重要**：EC8+2 需要最少 5 台服务器（才能形成纠删码条带）。

使用 Bash 工具计算，例如：
```bash
echo "scale=2; 需求容量TiB / (16 * 0.909) / 32 / 0.81 / 0.8" | bc
```
然后向上取整。

### 步骤 3：如果台数不足则降级到 EC4+2

如果 `所需台数 < 5`，则使用 EC4+2 重新计算：

```
所需台数 = 需求容量(TiB) / (单盘容量(TB) × 0.909) / 32 / 0.81 / 0.6667
```

其中：
- `0.6667`：EC4+2 可用率（66.67% 效率）

使用 Bash 工具重新计算。

**重要**：EC4+2 需要最少 3 台服务器。如果计算结果 < 3 台，则向上取整到 3 台。

### 步骤 4：检查性能需求

如果用户提供了性能需求，验证当前配置是否满足。

计算集群总性能：
```
总磁盘数 = 所需台数 × 32
上传带宽总计 = 总磁盘数 × 30 MiB/s
下载带宽总计 = 总磁盘数 × 60 MiB/s
上传 OPS 总计 = 总磁盘数 × 100
下载 OPS 总计 = 总磁盘数 × 300
```

将用户需求转换为 MiB/s 并比较：
- 如果是 Mbps：`MiB/s = Mbps / 8 × 0.9536`
- 如果是 Gbps：`MiB/s = Gbps × 1000 / 8 × 0.9536`
- 如果是 MB/s：`MiB/s = MB/s × 0.9536`
- 如果是 GB/s：`MiB/s = GB/s × 1000 × 0.9536`

使用 Bash 工具进行单位转换和比较。

### 步骤 5：如果性能不足则调整磁盘大小

如果任何性能指标未达标，尝试使用更小的磁盘以增加服务器数量（从而提升总性能）。

磁盘大小选择顺序（递减）：**16TB → 12TB → 10TB → 8TB**

对于每个更小的磁盘大小：
1. 使用当前纠删码方案重新计算 `所需台数`
2. 重新计算集群总性能
3. 检查是否所有性能需求都已满足
4. 如果满足，停止；如果不满足，尝试下一个更小的磁盘

使用 Bash 工具依次计算每种磁盘配置。

### 步骤 6：优化纠删码方案

如果最终配置满足：
- `所需台数 ≥ 5`，且
- 当前使用的是 EC4+2

则切换到 EC8+2（保持相同的服务器台数）以最大化可用容量：

```
最终容量(TiB) = 所需台数 × 32 × (单盘容量(TB) × 0.909) × 0.81 × 0.8
```

使用 Bash 工具重新计算最终容量。

### 步骤 7：计算并输出最终方案

输出最终配置方案：

**配置方案：**
- 服务器台数：`X` 台
- 纠删码方案：`EC8+2` 或 `EC4+2`
- 磁盘配置：每台服务器 `32 × Y TB HDD`

**容量：**
- 可用容量：`Z TiB`（同时显示为 TB、PB 或 PiB，根据规模选择合适单位）

**性能：**
- 上传带宽：`A MiB/s`（同时显示为 Gbps 或 GB/s）
- 下载带宽：`B MiB/s`（同时显示为 Gbps 或 GB/s）
- 上传 OPS：`C IOPS`
- 下载 OPS：`D IOPS`

## 计算技巧

- 使用 Bash 工具执行所有算术计算以确保精度
- 始终将服务器台数向上取整到最接近的整数
- 展示中间计算步骤，让用户理解计算逻辑
- 如果即使使用 8TB 磁盘也无法满足性能需求，告知用户并建议增加服务器数量或调整需求
- 以清晰、结构化的格式呈现最终方案

## 交互流程示例

1. 询问："您的容量需求是多少？（推荐使用 TiB 或 PiB 单位）"
2. 询问："您是否有性能需求？（可选：上传/下载带宽和 OPS）"
3. 计算满足容量所需的服务器台数
4. 如果存在性能需求，验证并在必要时调整磁盘大小
5. 如果适用，优化纠删码方案
6. 呈现最终方案及容量和性能预估

## 重要说明

- 始终将所有单位转换为统一基准（TiB、MiB/s）后再计算
- TB 到 TiB 转换：`TiB = TB × 0.909`
- MB/s 到 MiB/s 转换：`MiB/s = MB/s × 0.9536`
- EC8+2 提供更好的空间效率，但需要至少 5 台服务器
- EC4+2 需要至少 3 台服务器
- 更小的磁盘提供更高性能（更多磁盘轴数），但每台服务器容量更少
- 0.81 系数考虑了元数据、快照和系统开销
- 性能估算基于单盘平均性能，假设负载均匀分布
- 所有计算都使用 Bash 工具执行，确保数值精确
